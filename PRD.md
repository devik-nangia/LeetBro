## 1. Product Overview

### 1.1 The Problem
Platforms like LeetCode offer a binary learning experience: users either solve the problem or give up and look at the full solution code. This creates a "tutorial hell" loop where users read solutions without developing the underlying problem-solving mindset or algorithmic thinking required for technical interviews.

### 1.2 The Solution
**LeetBro** is an AI-powered learning companion for LeetCode. It acts as an intermediary step between being stuck and looking at the answer. By providing progressively helpful hints, conceptual approaches, algorithmic breakdowns, and visual aids generated by AI, LeetBro forces users to think critically. Progress is tracked manually via a "Mark as Solved" loop, and a prediction engine advises users on their readiness for specific problems based on their known DSA topics.

### 1.3 Target Audience
* Computer Science students preparing for internships/placements.
* Self-taught developers studying for technical interviews.
* Anyone struggling with the steep difficulty curve of raw LeetCode problems.

## 2. Objectives & Success Metrics

* **Primary Objective:** Build a fully functional, highly performant Next.js web application that seamlessly integrates unofficial LeetCode data with Google Gemini's generative capabilities.
* **System Objective:** Implement an aggressive caching strategy (PostgreSQL) to minimize expensive AI API calls and avoid LeetCode GraphQL rate limits.
* **User Success Metric:** Users should spend more time in the "Approach" and "Algorithm" tabs than the "Solution" tab.

## 3. Core User Flows

### Flow 1: Onboarding & Setup
1. User lands on the homepage and clicks "Sign In".
2. Authenticates via Google OAuth or Email/Password.
3. Directed to the **Topic Mastery Onboarding Screen**.
4. User is presented with a checklist of core DSA topics (e.g., Arrays, Hash Tables, DFS, BFS, Dynamic Programming).
5. User selects topics they are currently comfortable with.
6. Data is saved to the user's profile, and they are redirected to the main dashboard.

### Flow 2: Searching & Learning (The Core Loop)
1. User types a LeetCode problem title (e.g., "Merge Intervals") into the global search bar.
2. System fetches problem metadata (caches it if new) and renders the Problem View.
3. User navigates through the tabs:
    * *Hint 1* -> *Hint 2* -> *Hint 3* (Increasingly obvious clues).
    * *Approach* (Mindset discussion).
    * *Algorithm* (Step-by-step logic).
    * *Visualise* (Flowchart/diagram representation).
4. User writes code on the actual LeetCode website and submits.
5. Upon getting an "Accepted" status on LeetCode, the user returns to LeetBro and clicks **"Mark as Solved"**.
6. System logs the problem to the user's history and updates the sidebar.

## 4. Feature Requirements & Specifications

### 4.1 Global UI/UX
* **Theme:** Strict dark mode default. Background: `#1A1A1A`, Primary Accents: `#FFA116` (LeetCode Orange).
* **Layout:** Fixed left sidebar (250px width), fixed top navigation header, scrollable main workspace.
* **Component Library:** shadcn/ui for standardized, accessible inputs, dropdowns, dialogs, and tabs.

### 4.2 Sidebar (History & Navigation)
* **User Stats Header:** Displays total solved count.
* **Search/Filter:** Local text input to filter *only* the user's solved questions. Dropdown to filter by difficulty (Easy, Medium, Hard).
* **List View:** Chronological list of previously solved questions. Clicking one loads the cached question data into the main workspace.

### 4.3 Top Navigation Bar
* **Global Search:** A combobox/autocomplete input. Queries the backend for LeetCode questions.
* **Profile Dropdown:** Contains links to "Settings" (to update known DSA topics) and "Sign Out".

### 4.4 Main Workspace (The Problem View)
**Header Section:**
* Problem Title (e.g., "1. Two Sum").
* Difficulty Badge (Color-coded: Easy=Green, Medium=Yellow, Hard=Red).
* Tags (e.g., `Array`, `Hash Table`).
* **"Mark as Solved" Button:** Toggleable button. Confetti animation triggers on first click.

**Tabbed Interface (`<Tabs>` component):**

| Tab Name | Data Source | Acceptance Criteria |
| :--- | :--- | :--- |
| **Description** | LeetCode GraphQL | Must render raw HTML/Markdown from LeetCode cleanly. |
| **Hints (1, 2, 3)** | Gemini API (Cached) | Must be hidden by default. User must click "Reveal Hint X" to see it. Hint 3 must almost give away the algorithm without showing code. |
| **Approach** | Gemini API (Cached) | Text explaining *how* to think about the problem (e.g., "Since the array is sorted, we can optimize this using two pointers..."). |
| **Algorithm** | Gemini API (Cached) | Bulleted or numbered list of exact logical steps to implement the approach. Strictly NO code blocks allowed. |
| **Visualise** | Gemini API (Cached) | Renders a Mermaid.js diagram or a formatted text-based step-through of state changes. |
| **Solution** | Gemini API (Cached) | Contains the actual optimal code block in Python, Java, and C++. Includes a copy-to-clipboard button. |
| **Stats** | Internal Logic Engine | Compares Problem Tags with User's Known Topics. Renders an alert: "✅ You have the prerequisites" OR "⚠️ You are missing: [Topic Name]". |

## 5. Technical Architecture & Constraints

### 5.1 Tech Stack
* **Frontend:** Next.js (App Router), React, TailwindCSS, shadcn/ui, Lucide Icons.
* **Backend:** Next.js Route Handlers (`app/api`).
* **Database:** PostgreSQL.
* **ORM:** Prisma.
* **Auth:** Auth.js (NextAuth) or Clerk.
* **AI:** Google Gemini API (using structured JSON outputs).

### 5.2 The Caching Engine (Crucial System Design)
To ensure the app remains fast and API costs remain essentially zero, the backend must strictly follow this logic when a user searches for a question:
1.  **Check DB:** Query PostgreSQL for the question slug.
2.  **Cache Hit:** If found, return data instantly to frontend.
3.  **Cache Miss (Metadata):** If not found, call LeetCode GraphQL API. Save Title, Description, Tags, and Difficulty to DB.
4.  **Cache Miss (AI Content):** Trigger Gemini API with a strict system prompt to generate Hints, Approach, Algorithm, and Visualise data. Save to DB.
5.  Return newly constructed data object to frontend.

### 5.3 Data Fetching & State
* Use React Server Components (RSC) where possible for initial page loads.
* Use Client Components only for interactive elements (Tabs, Mark as Solved button, Search Input).
* State synchronization: When "Mark as Solved" is clicked, optimistically update the UI, then send a `POST` request to the backend.

## 6. Out of Scope (For V1)
* **Live LeetCode Syncing:** No scraping or session cookie hijacking. Relies on the user clicking "Mark as Solved".
* **Integrated Code Editor:** Users will write and execute code on the actual LeetCode platform, not within LeetBro.
* **Social Features:** No leaderboards or viewing other users' profiles.
